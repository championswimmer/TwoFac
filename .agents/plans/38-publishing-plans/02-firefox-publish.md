# Publishing TwoFac to Firefox Add-ons (AMO)

> Step-by-step guide for a first-time submission of the TwoFac browser extension to addons.mozilla.org.

## Extension Details

| Field | Value |
|---|---|
| Name | TwoFac |
| Add-on ID | `twofac@arnav.tech` |
| Manifest Version | 3 |
| Min Firefox Version | 128.0 |
| Summary | Two-factor authentication codes in your browser |
| Category | Security & Privacy |
| Permissions | `storage` |
| CSP | `wasm-unsafe-eval` (required for WebAssembly) |
| Build command | `./gradlew packageFirefoxExtension` |

## 1. Create a Mozilla / Firefox Account

- Sign up at <https://accounts.firefox.com/>
- Confirm your email; optionally add 2FA and a recovery key
- This same account is used for AMO — no separate "developer" registration

📖 [Submitting an add-on (account section)](https://extensionworkshop.com/documentation/publish/submitting-an-add-on/)

## 2. Accept the Developer Agreement

- Go to <https://addons.mozilla.org/developers/>
- Read and accept the **Firefox Add-on Distribution Agreement**
- Read the [Add-on Policies](https://extensionworkshop.com/documentation/publish/add-on-policies/)

## 3. Build the Extension ZIP

```bash
./gradlew packageFirefoxExtension
```

Verify the output ZIP contains at minimum:
- `manifest.json` (with `manifest_version: 3`, `browser_specific_settings.gecko.id` = `twofac@arnav.tech`, `browser_specific_settings.gecko.strict_min_version` = `128.0`)
- All `.wasm`, `.js`, `.html`, icon files
- Background scripts (Firefox MV3 uses `"background": { "scripts": [...] }`, not `service_worker`)

📖 [Package your extension](https://extensionworkshop.com/documentation/publish/package-your-extension/)
📖 [Manifest V3 migration guide](https://extensionworkshop.com/documentation/develop/manifest-v3-migration-guide/)

## 4. (Optional) Test Locally with `web-ext`

```bash
npx web-ext lint --source-dir <unzipped-dir>
npx web-ext run --source-dir <unzipped-dir> --firefox=firefoxdeveloperedition
```

📖 [web-ext command reference](https://extensionworkshop.com/documentation/develop/web-ext-command-reference/)

## 5. Submit to AMO

1. Go to **Add-ons Developer Hub** → [Submit a New Add-on](https://addons.mozilla.org/developers/addon/submit/)
2. Select **"On this site"** (listed on AMO)
3. Upload the ZIP → fix any validator errors (warnings about `wasm-unsafe-eval` CSP are expected)
4. Select compatible platforms → **Continue**
5. **Upload source code** — **required** (see §6 below)
6. Fill in listing details:

| AMO Field | What to Enter |
|---|---|
| **Name** | `TwoFac` |
| **Add-on URL** | `twofac` (→ `addons.mozilla.org/addon/twofac/`) |
| **Summary** | Two-factor authentication codes in your browser |
| **Description** | Longer description with feature list (supports TOTP/HOTP, imports from other apps, etc.) |
| **Category** | Security & Privacy |
| **Tags** | `2fa`, `totp`, `authenticator`, `otp` |
| **Icons** | 48×48 and 128×128 PNG icons |
| **Screenshots** | At least 1 screenshot of the popup/panel UI |
| **Homepage** | Link to project website or GitHub |
| **Support URL/email** | Contact info for user support |
| **License** | Select appropriate license |
| **Notes to Reviewer** | See §6 for what to write here |

7. Click **Submit Version**

📖 [Submitting an add-on](https://extensionworkshop.com/documentation/publish/submitting-an-add-on/)

## 6. Source Code Submission (REQUIRED for Wasm)

Firefox **requires** source code when the extension contains compiled/machine-generated code. Since TwoFac uses Kotlin/Wasm compiled to WebAssembly, this is mandatory.

### What to provide

Upload a ZIP/tarball of the **full repository source** that can reproduce the extension. Include:

- Complete Kotlin source code (`sharedLib/`, `composeApp/`)
- Gradle build files (`build.gradle.kts`, `settings.gradle.kts`, `gradle/`)
- The `gradlew` wrapper
- Any other files needed to build

### Build instructions (include in a `BUILD_INSTRUCTIONS.md` at the root of the source ZIP)

```
## Prerequisites
- JDK 21 (Temurin recommended)
- Node.js 22+

## Build
./gradlew packageFirefoxExtension

## Output
The Firefox extension ZIP is produced at:
composeApp/build/distributions/firefox-extension/twofac-firefox.zip
(adjust path to actual output location)
```

> ⚠️ **Lockfiles required**: Mozilla reviewers must reproduce the build exactly. Include `package-lock.json` (npm) or `yarn.lock` (yarn) at the root of the source ZIP. Missing lockfiles cause build mismatches and potential rejection.

### Notes to Reviewer (paste into the AMO reviewer notes field)

> This extension is built with Kotlin Multiplatform, compiled to WebAssembly.
> The `wasm-unsafe-eval` CSP directive is required to run the compiled Wasm module.
> Source code and build instructions are attached. Build requires JDK 21 and Node.js 22.
> Run `./gradlew packageFirefoxExtension` to reproduce the extension ZIP.
> The Wasm binary is deterministically generated by the Kotlin/Wasm compiler.

**Reviewers will rebuild from source and diff against your submitted ZIP** — ensure the build is reproducible.

## 7. Wasm-Specific Considerations

- **`wasm-unsafe-eval`**: Required in `content_security_policy.extension_pages` to instantiate WebAssembly modules. This will trigger a validator warning — it's expected and acceptable with source code provided.
- **Deterministic builds**: Kotlin/Wasm output should be reproducible given the same JDK + Kotlin compiler version. Pin tool versions in your build instructions.
- **Background scripts**: Firefox MV3 does **not** support `service_worker` in the `background` key. Use `"background": { "scripts": ["..."] }` instead. The `packageFirefoxExtension` Gradle task should handle this.
- **Side panel**: Firefox uses the `sidebar_action` API (not Chrome's `sidePanel`). Ensure the manifest uses the Firefox-compatible key.

📖 [CSP for extensions](https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/content_security_policy)
📖 [Browser compatibility for background](https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/background)

## 8. Review Process

- **Auto-approval**: Some extensions are auto-approved after passing automated validation. First-time submissions with Wasm will likely go to **manual review**.
- **Timeline**: Manual reviews typically take a few days to a few weeks. Complex extensions (Wasm, many permissions) may take longer.
- **Communication**: You'll receive email updates. Reviewers may ask follow-up questions about the Wasm build or request updated source.
- **After approval**: The extension is publicly listed on AMO. Future updates follow the same upload → review cycle.

## 10. Aug 2025 Policy Updates (New)

Mozilla published updated add-on policies effective **August 4, 2025**. Key changes relevant to TwoFac:

### Closed-group extensions now allowed
- Extensions for **internal or private use** among a small group are now permitted
- Useful if TwoFac needs enterprise/organizational distribution
- Previously prohibited — now lifts that restriction

### Implicit consent for self-evident single-use features
- If an extension's data transmission is **obviously required** for its functionality, explicit consent is implied
- Examples: image search, file uploaders
- For TwoFac: OTP secrets are stored locally via `storage` — no external transmission occurs, so this is not directly applicable but good to know

### Privacy policy hosting
- Privacy policies are **no longer required to be hosted on AMO**
- Self-hosted privacy policy links are acceptable
- Reduces update friction — no new version submission needed just for policy changes

### Source code dependencies
- All dependencies must be **included in the source ZIP** or downloaded **only via official package managers** during build
- No web-based build tools allowed (must run locally on the reviewer's machine)
- Reinforces the lockfile requirement in §6
